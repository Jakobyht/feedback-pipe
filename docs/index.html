One POST turns feedback into agent work.

A single local program on the machine where your coding agent lives. It receives a feedback message over HTTP and hands the user's words verbatim to your agent — no model, no judgement, no cloud. Your app's entire integration is one HTTP call.

How it works

Three parts, one direction

Your app sends one authenticated POST. The pipe writes the feedback into a prompt file and runs your agent in your repository — one agent at a time. Nothing ever flows back to the sender.

Your app

Sends one POST

The feedback message, with your PIPE_API_KEY as a Bearer token.

→
The pipe

Writes & queues

Saves the words verbatim to .ape/tasks/<id>/task.md and queues the run.

→
Your agent

Edits the code

Claude Code by default. It reads the prompt and decides everything.

The whole story, in plain words

  1. A user types feedback and clicks send.
  2. Your code makes one HTTP POST to the pipe. That is the only code you write.
  3. The pipe checks the key and answers 202 immediately.
  4. It writes the user's words, unchanged, into a prompt file inside your repo.
  5. It starts your agent in the repo — or queues the task if an agent is already running, because two agents editing the same checkout would corrupt each other's work.
  6. The agent reads the feedback and changes the code. The pipe never reports back.
The pipe contains no AI. It assigns no priority, invents no acceptance criteria, rewrites nothing. It is the wire, not the worker — every decision about the code belongs to the agent.
Before you start

What you need

Node.js 20+

The pipe is a single Node program with zero runtime dependencies.

Your coding agent

Installed on the same machine. Claude Code (the claude command) by default; any CLI agent works via PIPE_AGENT_COMMAND.

A local checkout of your repo

The repository the agent will edit. The pipe writes task files into .ape/ there (kept out of git automatically).

The tutorial

Step-by-step setup

Steps 1–3 run the pipe on the agent's machine. Step 4 is the one call your app adds. Step 5 proves the whole path works.

  1. Install the pipe

    Terminal on the agent's machine

    Clone and install. There is nothing to build.

    git clone https://github.com/Jakobyht/feedback-pipe.git
cd feedback-pipe
npm install

  2. Configure it

    Terminal — edit .env

    Copy the template and set the two required values. .env is per-machine and never committed.

    cp .env.example .env
    # .env
PIPE_API_KEY=make-a-long-random-secret
PIPE_REPO=/absolute/path/to/the/repo/the/agent/edits
    • The agent defaults to Claude Code headless. Override with PIPE_AGENT_COMMAND for Codex, Aider, or anything else.
    • The agent's own model key (e.g. ANTHROPIC_API_KEY) stays in the agent's environment — the pipe never sees it.

  3. Run it

    Terminal on the agent's machine 
    npm run pipe

    It prints Feedback pipe listening on http://127.0.0.1:8181. Check it is alive:

    curl -s http://localhost:8181/health

  4. Send feedback from your app

    Your app code — wherever feedback is already handled

    Add one HTTP call in the function your app already has for a submitted feedback message — your form handler, your /feedback route, your support endpoint. Only message is required.

The API — one endpoint

    
Everything your app can do, in one request.

    
POST /feedback
Authorization: Bearer <PIPE_API_KEY>
Content-Type: application/json

{
  "message":  "The checkout button does nothing on mobile",  // required
  "pageUrl":  "/cart",                                       // optional
  "repoHint": "frontend",                                    // optional
  "metadata": { "userId": "123" }                            // optional
}

    
    
      
  • 202 { taskId, status: "forwarded" } — accepted and queued for the agent.
    • 
      
  • 401 wrong key · 400 invalid JSON · 422 missing message · 413 body over 256 KB.
    • 
      
  • There is no status endpoint and no callback — the pipe is one-directional by design.

Every variable, and where it lives

    
Two places. The pipe's config stays on the agent's machine; your app holds only the key it sends.

    

      
VariableRequiredWhat it is

      
On the agent's machine — in the pipe's .env

      
PIPE_API_KEYyesThe secret callers must send. Any long random string.

      
PIPE_REPOyesAbsolute path to the repository the agent edits.

      
PIPE_AGENT_COMMANDnoThe agent to run. Defaults to Claude Code headless. Receives APE_TASK_PROMPT_FILE, APE_TASK_FILE, APE_TASK_ID, APE_REPO_PATH.

      
PIPE_PORTnoListen port (default 8181).

      
PIPE_HOSTnoBind address (default 127.0.0.1; 0.0.0.0 accepts remote calls).

      
PIPE_WORKSPACEnoA label attached to forwarded tasks (default default).

      
In your app's backend

      
PIPE_API_KEYyesThe same secret, sent as the Bearer token. Backend only — never in a browser.

    

    
Calling the pipe from another machine? Set PIPE_HOST=0.0.0.0 and put it
      behind HTTPS (Caddy, nginx, or a tunnel), because the key and feedback otherwise travel in cleartext. The pipe itself
      never changes — you are only adding transport in front of it.
